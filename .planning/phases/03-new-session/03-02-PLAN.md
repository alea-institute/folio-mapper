<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Boot resolver → localStorage session data | Reads/copies another session's persisted data into a new key |
| handleNewTab → window.open | Opens a new browser tab |
| Two tabs sharing an adopted identity | Risk of split-brain writes to the same key |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Tampering / Data Integrity | Cross-tab session clobber (split-brain) | mitigate | Auto-resume (D-07) COPIES the most-recent session into a freshly generated tabId and the current tab owns that new key (Pitfall 5). No two tabs ever share a tabId, so no tab can clobber another tab's namespaced session (D-04). |
| T-03-07 | Tampering / Injection | `window.open` target URL | mitigate | window.open receives `window.location.pathname + '?new=1'` ONLY — never a user-controlled or query-derived URL. No open-redirect/window-injection surface. Verified by acceptance grep. |
| T-03-08 | Information Disclosure | Auto-resume loads wrong session via tampered registry tabId | accept | Registry is UI metadata only, no auth/privilege; a tampered tabId loads that session's own data — user-error scope only (RESEARCH.md Security Domain). |
| T-03-09 | Repudiation / Data Integrity | updatedAt jitter making LRU/auto-resume pick the wrong session | mitigate | updatedAt is stamped only inside the debounced setItem onWrite path (Pitfall 3), tying recency to actual persistence; D-14 sort is deterministic. |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (RESEARCH.md Package Legitimacy Audit: zero new packages). No slopcheck checkpoint required. |
</threat_model>
