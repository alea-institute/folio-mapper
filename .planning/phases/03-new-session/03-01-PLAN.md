<objective>
Build the per-tab session storage core: a synchronous tab-identity resolver, a session
registry with LRU eviction, and an extension to the debounced storage adapter that updates
the registry on write. This is the foundation the stores and boot resolver consume in later
waves. No React, no UI — pure modules and Web APIs.

Purpose: Replace the single shared localStorage session with a per-tab namespaced model
(D-04) that survives browser close (D-13), migrates legacy data (D-06), and is bounded by
an LRU cap (D-09) keyed on last-modified time (D-14).
Output: tab-identity.ts, session-registry.ts, extended session-storage.ts, and two Wave-0
test files (tab-identity.test.ts, session-registry.test.ts).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL query → boot resolver | `?new=1` is attacker-influenceable (a crafted link) |
| Browser devtools → localStorage | Same-origin scripts/users can read/write session data and registry |
| localStorage quota → write path | Unbounded session growth could exhaust quota |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering / Injection | `history.replaceState` on `?new=1` | mitigate | replaceState receives `window.location.pathname` ONLY — never the query string or any user-controlled value. No open-redirect surface (no `window.open(url)` in this plan). Verified by acceptance criteria grep. |
| T-03-02 | Spoofing | `?new=1` param manipulation by a third party | accept | No server-side effect; only generates a fresh client-side tab id. Worst case is a clean session in the attacker's own tab — benign (RESEARCH.md Security Domain). |
| T-03-03 | Denial of Service | localStorage quota via unbounded session growth | mitigate | LRU cap MAX_SESSIONS=5 (D-09) evicts least-recently-modified sessions on write; QuotaExceeded writes are swallowed with console.warn (existing pattern). |
| T-03-04 | Information Disclosure | Registry/session data readable by same-origin scripts | accept | Pre-existing posture (localStorage was already used). Registry stores only UI metadata (dates, counts, filename) — no secrets, no auth tokens. No change in posture. |
| T-03-05 | Tampering | Legacy migration corrupts/loses existing in-progress work | mitigate | Migration is centralized in tab-identity.ts (single point, Pitfall 2); copies value before removing legacy key; D-06 test asserts no data loss. |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (RESEARCH.md Package Legitimacy Audit: zero new packages). No slopcheck checkpoint required. |
</threat_model>
