<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| "New" button → window.open | Opens a new browser tab from a UI control |
| Picker → registry mutation (delete) | User deletes session data via UI |
| App load → no forced gate | Auto-resume happens silently; picker is opt-in |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Tampering / Injection | "New" button + Start New → window.open | mitigate | The handler (session.handleNewTab from Plan 02) opens `window.location.pathname + '?new=1'` only — UI never passes a user-controlled URL to window.open. No open-redirect surface. |
| T-03-11 | Denial of Service / Data Integrity | Picker Delete removes wrong/all session data | mitigate | deleteFromRegistry targets a single tabId from the rendered row and removes only that session's two namespaced keys; current tab is highlighted so the user sees which is active. No bulk-delete. |
| T-03-12 | Information Disclosure | Picker displays session metadata | accept | Metadata (dates, counts, filename) is non-sensitive UI data already stored client-side; no auth tokens or PII surfaced beyond what the user themselves entered. |
| T-03-13 | Spoofing / UX | Removing the recovery gate could silently load a stale/wrong session | mitigate | Auto-resume deterministically picks the most-recently-modified session (D-14); the on-demand picker (D-07b) lets the user switch/inspect/delete, and the current session is highlighted. |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (RESEARCH.md Package Legitimacy Audit: zero new packages). No slopcheck checkpoint required. |
</threat_model>
